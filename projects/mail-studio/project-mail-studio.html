Designing a modern workflow for Outlook email production

Mail Studio is an internal workflow tool built to simplify the creation and delivery of Outlook-compatible email templates and reusable email assets.

Overview

The project started inside a B2B marketing agency where enterprise clients frequently requested Outlook emailers and OFT [Outlook file template] deliverables. While the design workflow had evolved with tools like Figma, the final production process still depended heavily on Microsoft Word and manual table-based formatting. Designers were still rebuilding layouts manually just to make them work in Outlook.
Mail Studio was created to reduce that friction.
What began as a self-initiated experiment eventually evolved into an internal workflow system actively used by the design team to deliver production emailers for enterprise IT clients.

The Challenge

One recurring type of project at the agency involved creating Outlook email templates that clients could later reuse themselves. The workflow was surprisingly outdated.

A typical process looked like this: 

Design the email in Figma  → Slice sections into images  → Rebuild the layout manually inside Microsoft Word using tables → Copy everything into Outlook →  Export the final file as an OFT

The process was slow, repetitive, and fragile.

Even small edits often caused:
alignment issues, spacing inconsistencies, rendering bugs, broken layouts, text formatting problems

The issue wasn't designing the email. The issue was translating modern design output into a format Outlook could reliably render. And honestly, Every additional handoff—from Figma to Word to Outlook—introduced friction, inconsistency, and opportunities for things to break. That frustration became the starting point for Mail Studio.

Key Objectives

Reduce dependency on Microsoft Word by creating a more reliable and designer-friendly workflow for Outlook email production while improving overall production efficiency.

Old Workflow

Designers used to copy the entire design from figma or phtoshop and rebuilt it inside of word using a standard table.

Exploring Existing Tools

Before building the product, I explored several existing email builders including:
Stripo  |  Beefree.io  |  Canva | Zoho Campaigns | Some Figma Plugins

Many of these had very advanced features like
- ESP integrations, campaign management, newsletters,  marketing automation, collaborative email workflows

Some also came with pricing structures and usage restrictions that didn't make sense for occasional agency production workflows. More importantly, none of them solved the actual delivery problem we had.

Most html tools exported HTML mailers, pdfs, or images only. But our workflow required reusable Outlook-compatible files that clients could download, edit, and reuse independently without connecting to external platforms or ESPs. The issue wasn't "creating emails."

The issue was creating production-ready Outlook-native assets reliably.

The Initial Idea

The first vision for the product was much bigger than what eventually shipped. The original idea included:

  • advanced visual editing
  • drag-and-drop layouts
  • flexible canvas interactions
  • rich styling controls
  • full OFT export

Like most early product ideas, it started oversized.

The first usable version was intentionally reduced to the essentials.

  1. a block-based canvas
  2. text blocks
  3. image blocks
  4. spacers and dividers
  5. basic rich text editing
  6. simple layout controls
  7. EML export support

The editor worked by converting content blocks into HTML tables optimized for Outlook rendering. At the time, the rendering architecture itself became one of the biggest challenges. The early versions generated multiple stacked HTML tables instead of a cleaner unified structure, which caused layout inconsistencies and export issues.

A large part of the project quickly became understanding how Outlook actually behaves as a rendering environment.

Learning Outlook the Hard Way

One of the earliest goals for Mail Studio was direct OFT export. While it remained a highly requested capability, the initial focus shifted toward EML export to provide users with a reliable and efficient way to work with Outlook templates. This allowed the team to validate workflows, gather feedback, and improve the overall experience before tackling more complex export requirements. As the product matured, OFT export was successfully introduced, giving users greater flexibility while maintaining a seamless template creation process.

One of the hardest parts of the project was understanding Outlook itself. Modern frontend assumptions break very quickly inside Outlook environments.

Simple things like:

  • spacing
  • responsiveness
  • image rendering
  • text behavior
  • padding
  • alignment

…became implementation challenges.

Image handling became particularly difficult. Over time, the app moved toward more Outlook-safe rendering approaches involving:

  • VML-based rendering
  • embedded assets
  • CID-style image handling
  • Base64 embedding strategies

At the time, I didn't fully understand many of these systems deeply from an engineering perspective. A large part of the process involved:

  • researching unfamiliar concepts
  • experimenting rapidly
  • validating outputs
  • asking engineers for feedback
  • using AI-assisted technical reasoning
  • repeatedly testing inside Outlook

The project became as much about learning technical constraints as building the product itself. 

Building with AI-Assisted Development

I came into this project with limited engineering experience.

Because of that, the app was intentionally built using: Vanilla HTML | CSS | JavaScriptAlongside AI-assisted development tools including:

  • Claude
  • Codex
  • Antigravity
  • Google Stitch

The goal was to keep the system lightweight and understandable while still allowing fast experimentation.

AI-assisted development significantly accelerated prototyping and iteration. It reduced the gap between product thinking and implementation and allowed me to test workflows much faster than I could have traditionally.

At the same time, it also exposed limitations. 

Some early implementation decisions created long-term architectural problems:

  • rendering inconsistencies
  • layout behavior issues
  • image handling complexity
  • unstable padding logic

Looking back, I would choose a different stack entirely if rebuilding the product today. But the experience fundamentally changed how I think about prototyping and product creation as a designer.

The Figma Plugin

Even after building the first version of Mail Studio, one major inefficiency still remained.

Designers were still:

  1. designing inside Figma
  2. then manually recreating layouts again inside Mail Studio

The workflow duplication was still there.

That led to the development of a companion Figma plugin.

The plugin allowed designers to transfer layouts directly into the web app instead of rebuilding them manually.

The workflow worked through structured exports:

  • designers used vertical auto-layouts
  • predefined template frames improved consistency
  • layer-name prefixes controlled block behavior
  • unsupported layouts could automatically rasterize into images
  • export warnings informed users which sections would be rasterized

Designs could then be:

  • copied directly into Mail Studio as editable blocks
  • or exported as reusable .oftprj project files

One of the more difficult technical challenges was making the plugin intelligently interpret layouts when prefixes weren't used. Early versions rasterized too many frames because the system struggled to identify the closest editable block type reliably.

Over time, the export logic improved significantly.

The plugin eventually became one of the most important parts of the product because it reduced the biggest friction point: recreating layouts across tools.

The Figma plugin cutdown the production time by more than 50%

The final files were exported in eml format that can work across multiple email clients and ESPs

UX Challenges

The early versions of Mail Studio focused heavily on functionality.

Usability came later. That created a lot of friction initially.

Users expected the product to behave like Figma, Canva and other free form editors

But Mail Studio was intentionally structured as a block-based system because Outlook rendering required predictability. That mismatch created confusion.

Early versions also had:

  • minimal onboarding
  • no tooltips
  • overloaded controls
  • unclear interaction patterns

People struggled to understand how the system was supposed to work. Users found the system confusing because:

  • there were too many exposed controls
  • onboarding was minimal
  • there were no tooltips
  • interaction patterns weren't obvious
  • expectations didn't match the product structure

Versions 6 and 7 introduced major usability improvements including:

  • a redesigned UI
  • improved editing flows
  • clearer controls
  • HTML editing support
  • reusable templates
  • reusable block libraries
  • richer import workflows

The product gradually evolved from a simple export utility into a more complete workflow system.

Product Evolution

Over multiple iterations, Mail Studio expanded far beyond the original scope. New capabilities gradually included:

  • HTML editing
  • HTML import
  • reusable block systems
  • reusable templates
  • local/offline persistence
  • drag-and-drop improvements
  • richer import workflows
  • improved rendering behavior
  • bundled architecture cleanup
  • project file support (.oftproj)
  • direct .oft export

The product matured through constant iteration rather than long planning cycles.

One interesting realization was that users cared deeply about continuity and recovery.

Features like:

  • reusable blocks
  • reusable templates
  • local persistence
  • offline support

…became unexpectedly important in daily usage.

The product slowly evolved from "an Outlook export tool" into something closer to a lightweight production environment for email workflows.

Outcomes

Mail Studio is currently used internally by the design team as part of the agency workflow. The tool has been used to deliver multiple Outlook emailers and reusable email assets for large enterprise IT clients. 

More importantly, it significantly reduced dependence on Microsoft Word as part of the production process. The project also changed how I think about design itself.

Instead of focusing only on interfaces, the project forced me to think more deeply about:

  • systems
  • workflows
  • interoperability
  • operational constraints
  • adoption friction
  • technical limitations
  • production efficiency

One of the biggest realizations from the project was understanding how many inefficient workflows continue to exist simply because people have adapted around them over time.

Reflections [There is quite a lot!]

Reduce before you build. The TypeScript failure taught me to match the complexity of the tool to my ability to debug it. Simplifying to vanilla JS wasn't a compromise — it was the right decision.

Break the problem into smaller ones. When I wanted to add padding controls I didn't build the whole system at once — I started with global settings, tested it, then extended to individual blocks. Smaller steps gave me better decisions at every stage.

Test with real users immediately. Both critical feedback rounds — the designer's UI feedback and the manager's rendering insight — revealed things I couldn't see myself because I was too close to the build.

AI-assisted testing is a real workflow. Using Manus for recurring feature validation before human testing is faster, cheaper, and catches obvious issues early. Also the same AI coding agent helped me understand how the code works, just by asking it "What does this section in the code do?"

Match the model to the task. I started building everything with Claude Opus and burned through tokens faster than I could track. Lighter models for planning and simple fixes, heavier models only when the problem actually needs them.

Switching models mid-workflow can break things. I switched to Gemini for refactoring the Figma plugin and it introduced syntax Figma doesn't support. Finish what you start with what you started with, or validate carefully when switching.

Read the implementation plan even if you can't read the code. The plan an AI IDE generates before writing code tells you what it's building — even without understanding the code itself, you can catch wrong assumptions early.

Learn the basics yourself. I picked up basic Git so I could push to GitHub without relying on AI for every upload. Some things are worth knowing yourself.

You don't need to code to build products. You need to understand the problem deeply enough to know what the solution has to do — and the stubbornness to keep fixing it until it works.
